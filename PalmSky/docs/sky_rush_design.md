# 飞升小游戏设计文档：冲九霄

## 1. 目标

为最终飞升阶段补齐第四个大境界小游戏，替换当前 `skyRush` 占位实现。

设计目标：
- 终局感要强，和“羽化飞升”语义一致
- 适配 `Apple Watch` 小屏，不依赖复杂手势
- 与前 3 个小游戏形成能力闭环，不重复
- 能直接接入现有 `MiniGameContainer` / `GameLevelManager` / `finalizeMiniGame`


## 2. 现有小游戏规律

当前已存在的 3 个小游戏：

1. `mindDemon`：破心魔
   - 核心能力：单点时机
   - 输入：点击发射
   - 表现：静态中心 + 动态障碍

2. `swordDefense`：御剑挡劫
   - 核心能力：反应与方向切换
   - 输入：点击切换旋转方向
   - 表现：中心防守

3. `inscription`：推演阵法
   - 核心能力：观察与记忆
   - 输入：按顺序点击
   - 表现：阵法复刻

它们的共同特点：
- 规则简单，2 秒内能理解
- 输入方式单一
- 单局时长短
- 失败原因明确

第四个小游戏必须延续这四点。


## 3. 终局方案结论

最终飞升小游戏采用：

**玩法名：`冲九霄`**

核心形式：
- 三轨自动上升
- 玩家通过左右半屏点击切换轨道
- 躲避障碍并收集 `升仙印`
- 达成“存活到结束”或“收集目标数”即成功

这不是传统跑酷，而是适配手表的 **三轨登天**。


## 4. 为什么不用传统跑酷

不采用自由跳跃 / 下滑 / 连续滑动跑酷，原因如下：

- `watchOS` 屏幕太小，滑动容错低
- 手表更适合单点输入，不适合复杂连续手势
- 传统跑酷对横向空间要求高，画面会乱
- 你现有 3 个小游戏都不是“连续高输入负担”设计

因此最终方案只保留：
- 左半屏点击：向左换道
- 右半屏点击：向右换道

这是当前最稳的交互方案。


## 5. 玩法规则

### 5.1 基础结构

- 画面存在 `左 / 中 / 右` 三条天阶轨道
- 主角（元神）默认处于中轨
- 背景持续向下卷动，制造“向上飞升”的视觉效果
- 障碍物和资源从屏幕上方向下进入

### 5.2 玩家操作

- 点击左半屏：若当前不在最左，则左移一轨
- 点击右半屏：若当前不在最右，则右移一轨
- 不做长按
- 不做滑动
- 不做跳跃

### 5.3 胜利条件

满足任一条件即可胜利：

1. **存活到倒计时结束**
2. **收集满指定数量的 `升仙印`**

这是双重判定设计：
- 稳健派可以专心活到结束
- 激进派可以主动吃印提前过关

### 5.4 失败条件

- 初始有 `2` 层护体罡气
- 第一次撞击：护盾破碎
- 第二次撞击：飞升失败

不使用红心 UI，统一用护体罡气表现。


## 6. 视觉与修仙包装

### 6.1 主角

- 一个金色元神光点 / 小型人形光影
- 外层包裹一圈青金色护体罡气

### 6.2 护体罡气

- 初始状态：可见球形护盾
- 受击一次：护盾破碎并消失
- 无护盾时：角色裸奔状态，危险感更强

### 6.3 障碍物

建议至少两类：

1. `天雷`
   - 单轨基础障碍
   - 视觉清晰，识别成本低

2. `罡风裂隙`
   - 单轨或双轨短暂封路
   - 用于制造强制换道

### 6.4 收集物

`升仙印`
- 金色 / 白金色
- 收集时给清脆反馈
- 用于激进打法的目标物

### 6.5 成功结尾

必须有终局镜头：
- 天门开启
- 元神冲入白光
- 屏幕短暂纯白
- 再进入飞升成功结算

如果缺这段，终局情绪不够。


## 7. 难度设计

### 7.1 对应区间

`skyRush` 用于最终四个大境界突破：
- 天仙
- 真仙
- 玄仙
- 九天玄仙

也就是 `GameLevelManager.getTribulationGameType(for:)` 中的最后一段。

### 7.2 难度维度

难度只通过这几个参数提升：
- 上升速度
- 障碍生成频率
- 双障碍出现概率
- 目标收集数

### 7.3 建议参数

#### 第一档（天仙）
- 时长：16 秒
- 目标：收集 6 个升仙印
- 障碍节奏偏松

#### 第二档（真仙）
- 时长：18 秒
- 目标：收集 7 个升仙印
- 开始出现连续换道

#### 第三档（玄仙）
- 时长：20 秒
- 目标：收集 8 个升仙印
- 开始出现双障碍封路

#### 第四档（九天玄仙）
- 时长：22 秒
- 目标：收集 10 个升仙印
- 速度最快，压迫感最强


## 8. 与现有代码的接入方式

### 8.1 `GameLevelManager`

当前定义中已经有：
- `TribulationGameType.skyRush`

但当前映射仍然是：
- `case 12...15: return .mindDemon`

需要调整为：
- `case 12...15: return .skyRush`

对应文件：
- `PalmSky Watch App/manager/GameLevelManager.swift`

### 8.2 `MiniGameContainer`

当前 `.skyRush` 只是文本占位：

- `Text(NSLocalizedString("watch_game_skyrush_dev", comment: ""))`

需要替换为正式视图 / 场景：
- `SkyRushGameView`
或
- `SkyRushScene + SpriteView`

对应文件：
- `PalmSky Watch App/view/game/MiniGameContainer.swift`

### 8.3 结算

现有结算逻辑已经通用：
- `handleMiniGameFinish(isWin:)`
- `GameManager.finalizeMiniGame(isWin:)`

因此新小游戏只需要在结束时回调：
- `onFinish(true)`
- `onFinish(false)`

不需要额外改结算规则。


## 9. 技术实现建议

### 9.1 选型

建议使用 `SpriteKit`，理由：
- 现有 `mindDemon` / `swordDefense` 已使用 `SpriteKit`
- 轨道切换、障碍生成、滚动背景都适合场景驱动
- 易于做飞升终局特效

### 9.2 节点结构

建议节点：
- `backgroundNode`
- `laneGuideNodes`
- `playerNode`
- `shieldNode`
- `collectibleLayer`
- `hazardLayer`
- `topUI`

### 9.3 核心状态

需要的状态：
- 当前轨道 `currentLane`
- 剩余护盾层数 `shieldCount`
- 已收集升仙印数量 `sealCount`
- 已进行时间 `elapsed`
- 游戏是否结束 `gameEnded`

### 9.4 生成逻辑

建议每次生成一“波”内容，而不是完全随机刷：
- 单个障碍
- 单个升仙印
- 障碍 + 升仙印诱导
- 双轨封路
- 全轨安全奖励波

这样节奏会更像设计过的终局关，而不是无脑随机。

## 11. 设计边界

以下内容暂不做：
- 滑动操作
- 长按蓄力
- 连击系统
- 多技能系统
- 手表体感控制
- 多阶段 Boss

原因：
- 这些会明显增加输入复杂度
- 不符合现有小游戏“短、直、准”的设计标准


## 12. 最终定稿

第四个小游戏定为：

**`冲九霄（三轨登天）`**

最终特征：
- 三轨换道
- 双重胜利条件
- 护体罡气两命
- 终局飞升镜头
- 完全适配 Apple Watch 小屏操作

这是当前最适合《掌上修仙》终局飞升的方案。
