# System Design

本文档记录当前项目的核心系统设计，优先覆盖正在开发与高频维护的游戏机制。

## 1. 闭关系统

### 系统目标

- 为玩家提供“离线也在修炼”的持续反馈。
- 将 Apple Watch 的碎片化使用方式转化为修仙题材中的“闭关”体验。
- 在不要求实时在线的情况下，维持数值成长与回归动机。

### 核心规则

- 玩家进入后台时记录 `lastLogout`。
- 玩家回到前台后，根据离线时长结算闭关收益。
- 离线时间存在阈值与上限:
  - 小于 5 分钟不结算。
  - 免费版与付费版有不同离线收益上限。
- 离线收益基于当前境界的自动修炼效率计算，并进行离线折算。

### 主要职责归属

- `GameManager.calculateOfflineGain()`
  - 计算离线时长。
  - 应用离线收益规则。
  - 产出 Toast 文案。
  - 刷新 `lastLogout` 并持久化。
- `RootPagerView`
  - 监听 `scenePhase`。
  - 在前后台切换时触发刷新与结算。

### 设计原则

- 闭关收益必须稳定、可解释，避免玩家通过频繁切前后台刷收益。
- 收益结算必须与本地存档绑定，云端只做备份，不做实时状态源。

## 2. 睡眠系统

### 系统目标

- 将“昨夜睡眠”融入修仙语境，构建“闭关养神”玩法。
- 强化回归奖励的仪式感，提高次日打开应用的动机。
- 形成轻量健康联动，而不是把产品做成健康 App。

### 当前设计

- 数据来源: HealthKit 睡眠分析数据。
- 查询窗口: 昨日夜间至今日中午的睡眠记录。
- 结果映射:
  - 低于 5.5 小时: 无额外加成。
  - 5.5 到 6.5 小时: 低档加成。
  - 6.5 到 7.5 小时: 中档加成。
  - 7.5 小时及以上: 高档加成。
- 生效时机:
  - 仅在“今日首次有效离线收益结算”时生效。
  - 每日只消费一次，避免反复切后台刷奖励。

### 当前状态

- 当前判断: 睡眠系统核心链路已基本完成，可进入真机验证与回归测试阶段。
- 当前已完成的关键问题修复:
  - 睡眠查询与离线结算的时序竞争已修复。
  - 睡眠数据不可用时的误导性展示已修复。
  - 睡眠 sample 直接累加导致的重叠双算风险已修复。
  - 设置页已进一步区分“未授权”和“无数据”，避免继续混用同一状态文案。

### 当前已实现逻辑

- 权限读取:
  - 已从“仅步数”扩展到“步数 + 睡眠”读取权限。
- 数据抓取:
  - 应用启动时请求并拉取睡眠数据。
  - 回到前台时再次刷新睡眠数据。
- 结果展示:
  - 设置页已展示昨夜睡眠时长、养神档位、离线加成和今日状态。
  - 睡眠状态展示当前采用产品态双态区分:
    - 尚未进入睡眠授权流程时显示“未授权”。
    - 已请求过权限但暂未查到睡眠样本时显示“无数据”。
- 奖励结算:
  - 离线收益在基础闭关收益上叠加睡眠倍率。
  - 睡眠奖励只在当天首次有效离线结算中消费一次。
- Buff 联动:
  - 中后期境界在满足睡眠档位条件时可获得短时自动修炼 Buff。
  - Buff 状态栏与 Buff 详情页已支持睡眠来源样式区分。

### 当前保留的取舍

- 睡眠 Buff 的额外提示文案暂不追加到主 Toast，避免提示过长。
- 暂不引入复杂的睡眠状态机，只保留“未授权/无数据”的最小修正。
  - 当前已从合并态升级为最小双态区分：未授权 / 无数据。
- 暂不调整睡眠查询窗口与睡眠奖励消费时机，接受“玩家在上午较早时间打开 App 时，睡眠统计可能偏少”的现阶段取舍。
- 暂不做睡眠参数配置化，后续如进入调平衡阶段再统一抽离。

### 境界联动

- 中后期境界在满足睡眠档位条件时，可额外获得短时自动修炼 Buff。
- 此 Buff 属于轻联动奖励，不应替代主养成循环。

### 主要职责归属

- `WatchHealthManager`
  - 请求 HealthKit 权限。
  - 拉取昨夜睡眠时长。
  - 提供睡眠档位、倍率和展示文案。
- `GameManager`
  - 管理睡眠奖励是否已被当日消费。
  - 在离线结算中应用睡眠倍率。
  - 在合适境界发放睡眠专属 Buff。
- `SettingsView`
  - 展示昨夜睡眠、档位、加成、今日状态。

### 后续待补强方向

- 处理“无权限 / 无数据 / 数据未同步完成”的明确状态区分。
- 保证睡眠查询完成后再触发离线收益结算，避免时序竞争。
- 统一睡眠 Buff 的提示文案与玩家可见反馈。
- 评估睡眠 sample 重叠累计问题，避免多数据源导致睡眠时长重复计算。
- 统一睡眠系统的前后台刷新策略，减少重复查询与不必要调用。

### 当前确认的优化清单

- 高优先级:
  - 已修复: 将“拉取睡眠数据”和“计算离线收益”串成确定性顺序，不再依赖固定延迟后结算。
  - 已优化: 睡眠时长统计改为“裁剪窗口 + 合并重叠区间 + 求总时长”，降低边界漏算与重叠双算风险。
- 中优先级:
  - 已采用最小修正:
    - 当睡眠数据不可用时，统一显示为“未授权/无数据”。
    - 不再把不可用状态误展示为“养神不足”或“+0%”。
  - 如后续有需要，再升级为更细的三态或四态展示。
- 低优先级:
  - 将睡眠档位、倍率、触发境界等配置进一步常量化，减少规则散落在业务实现中。

### 当前测试重点

- 不同睡眠档位下，设置页展示与离线结算倍率是否一致。
- 同一天首次有效离线结算后，睡眠奖励是否只消费一次。
- 未授权或无睡眠数据时，设置页是否稳定显示“未授权/无数据”。
- 早晨回前台时，睡眠查询完成后再触发离线结算的链路是否稳定。

## 3. iCloud 备份 / 恢复系统

### 系统目标

- 解决删 App、换设备、重装后丢档的问题。
- 为玩家提供“修炼进度有兜底”的安全感。
- 保持“本地为主、云端兜底”的简单模型，降低状态复杂度。

### 当前设计

- 存储载体: `NSUbiquitousKeyValueStore`。
- 备份策略:
  - 本地保存仍是主存档。
  - 云端保存为延迟写入的备份快照。
  - 进入后台时主动冲刷待写入云备份。
  - 应用启动时先同步一次 KVS，再尝试读取云端备份。
  - 后台冲刷后主动调用 `synchronize()`，提高写入及时性。
- 恢复策略:
  - 仅在本地无存档时尝试读取云端备份。
  - 如果存在云端备份，弹出提示让玩家决定是否恢复。
- 删除策略:
  - 重置游戏时同步删除本地存档和云端备份。

### 当前状态

- 当前判断: iCloud 备份 / 恢复主链路已基本完成，可进入真机恢复链路验证阶段。
- 当前已完成的关键问题修复:
  - 应用启动时先同步 KVS，再尝试读取云备份。
  - 应用进入后台前，主动冲刷并同步云端备份。

### 设计边界

- 不是实时云同步。
- 不做多端同时在线的合并冲突处理。
- Watch 端和 iPhone 端使用各自独立的 iCloud 备份 key。
- 暂不监听 `NSUbiquitousKeyValueStoreDidChangeExternallyNotification`，保持“本地优先 + 云端兜底”的简单模型。

### 主要职责归属

- `GameManager`
  - 读取云备份。
  - 管理恢复确认流程。
  - 节流写入 iCloud。
  - 重置时清空云端备份。
- `MainView`
  - 承接云端恢复确认弹窗。
- `SkyConstants`
  - 定义不同平台的云备份 key。

### 当前测试重点

- 本地无存档且云端有备份时，是否稳定弹出恢复提示。
- 点击“恢复云端”后，是否正确恢复境界与主存档。
- 点击“重新开始”后，本地是否正常新开存档，并按现有产品语义继续覆盖云端。
- 删档重置后，云端备份是否一并清除。

## 4. 步数收益系统

### 系统目标

- 让现实世界的步数转化为游戏内修炼资源。
- 强化 Apple Watch 的设备特性，形成差异化核心玩法。
- 鼓励轻运动，但不让其压过主修炼循环。

### 当前设计

- 数据来源: HealthKit 当日步数。
- 步数收益公式:
  - 将“可炼化步数”乘以“当前单次点击收益”得到灵气收益。
- 限制机制:
  - 每日有炼化上限。
  - 免费版与付费版有不同步数上限。
  - 已炼化步数按天重置。

### 主要职责归属

- `WatchHealthManager`
  - 拉取今日步数。
  - 管理每日炼化上限与已炼化步数。
  - 执行步数炼化。
- `GameManager`
  - 提供当前点击收益。
  - 触发炼化动画与主界面反馈。
- `SettingsView` / 相关组件
  - 展示步数炼化入口与结果反馈。

### 设计原则

- 步数收益应服务于“现实行为映射修炼行为”的主题表达。
- 步数收益必须有上限，避免极端行为破坏平衡。
- 需要兼顾免费体验和付费扩展空间。

## 5. 手动数据覆盖系统

### 系统目标

- 在不推翻“双端独立”架构的前提下，为玩家提供可控的进度校准手段。
- 解决手机与手表长期分开游玩后，进度分叉难以收敛的问题。
- 保持护身符与设备设置的本地独立边界。

### 当前设计

- 当前已支持“手表覆盖手机”与“手机覆盖手表”两种手动覆盖方向。
- 双端均通过 `WatchConnectivity` 即时消息主动发起覆盖请求。
- 目标端收到请求后，将来源端修炼进度写入自己的本地主存档。
- 护身符库存、设备设置与本地存档 `id` 在目标端保持本地值。
- `lastLogout` 在覆盖时重置为当前时间，避免异常离线收益。

### 设计边界

- 不是实时双向同步。
- 不做整包 `Player` 直接替换。
- 不做护身符跨端共享。
- 不做设置跨端共享。
- 保持“玩家主动发起 + 即时覆盖 + 本地落盘”的校准模型，不引入自动合并。

### 主要职责归属

- `SkySyncManager`
  - 发起双端手动覆盖请求。
  - 使用即时消息返回成功或失败结果。
- `GameManager`
  - 在目标端按白名单字段应用远端进度。
  - 覆盖完成后保存本地主存档并继续走本端备份链路。
- `SettingsView`
  - 按平台提供手表端或手机端覆盖入口、确认弹窗与结果反馈。
